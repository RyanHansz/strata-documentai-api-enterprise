# DocumentAI API

Serverless document processing platform with intelligent extraction, multi-tenant API key management, and an admin console.


> ⚠️ **Public Preview / Active Development**
> This project is under active development. APIs, configurations, and features are subject to change. 
> It is **not ready for production**.
> 
> While the core architectural foundation is set, features and APIs remain fluid as we harden the 
> codebase and complete the documentation.


## Repository Structure

```

├── admin-ui/           Admin console (vanilla JS SPA)
├── infra/              Terraform infrastructure
├── docs/               Architecture diagrams and specs
├── .github/            CI/CD workflows
```

## Prerequisites

- AWS CLI configured (`aws configure --profile your-profile`)
- Terraform 1.10+
- Docker
- Node.js 18+
- Python 3.11+ with [uv](https://github.com/astral-sh/uv)

## First-Time Setup

1. **Configure AWS profile**:

   ```bash
   export AWS_PROFILE=your-profile
   ```

2. **Bootstrap Terraform state bucket** (once per environment):

   ```bash
   cd infra
   make infra-bootstrap ENVIRONMENT=dev
   make infra-init ENVIRONMENT=dev
   ```

3. **Deploy infrastructure + UI**:

   ```bash
   cd ..
   make deploy
   ```

4. **Admin UI config** is auto-generated by `make deploy` (writes `admin-ui/config.json` from Terraform outputs). For local UI development without a prior deploy, copy manually:

   ```bash
   cp admin-ui/config.example.json admin-ui/config.json
   # Edit with your API endpoint and Cognito IDs
   ```

## Deploy

```bash
make deploy            # Everything (infra + UI)
make deploy-infra      # Build + push Docker image + Terraform apply
make deploy-ui         # Build admin UI bundle + S3 sync + CloudFront invalidation
```

Override defaults: `make deploy ENVIRONMENT=dev AWS_PROFILE=my-profile`

> **Note**: Region is configured in `infra/environments/<env>/terraform.tfvars`. State bucket is created in `us-east-1` by default.

### Teardown

```bash
cd infra
make infra-destroy ENVIRONMENT=dev
```

## Local Development

### API (documentai-api)

```bash
cd documentai-api
cp local.env.example .env    # First time only
make init                    # Build Docker containers
make start                   # Start API at localhost:8000
```

Or without Docker:

```bash
make init-local
export RUN_CMD_APPROACH=local
make start-local
```

### Admin UI

```bash
cd admin-ui
npm install
cp config.example.json config.json   # Edit with local API URL
npm run dev                          # Dev server at localhost:3000
```

## Components

### [documentai-api/](documentai-api/)

FastAPI application deployed as a Lambda container. Handles:

- Document upload and processing via Bedrock Data Automation (BDA)
- Multi-tenant API key authentication
- Extraction rule configuration
- Metrics collection and aggregation

```bash
cd documentai-api
make start        # Local development server (Docker)
make test         # Run pytest suite
make lint         # Ruff + mypy
```

### [admin-ui/](admin-ui/)

Single-page admin console for managing the platform. Features:

- API key lifecycle (create, revoke, audit)
- Tenant and user management with role-based access
- Extraction rule editor with field search
- Document viewer with detail panel
- Cognito auth with TOTP MFA + forgot-password

```bash
cd admin-ui
npm run dev       # Dev server at localhost:3000
npm test          # 267 unit tests (Vitest)
npm run test:e2e  # 17 e2e tests (Playwright)
```

### [infra/](infra/)

Terraform modules for the complete serverless stack. See [infra/README.md](infra/README.md) for full module list and usage.

```bash
cd infra
make infra-plan   # Terraform plan
make infra-apply  # Terraform apply
```

Only one environment exists currently (`dev`). To add another, copy `infra/environments/dev/` and update `terraform.tfvars`.

## Architecture

See [docs/documentai-api/diagrams/architecture.mmd](docs/documentai-api/diagrams/architecture.mmd) for the full system diagram.

### High-Level Flow

1. Client uploads document via API Gateway → Lambda
2. Document stored in S3, metadata in DynamoDB
3. EventBridge triggers document processor Lambda
4. Processor invokes BDA with tenant-specific blueprint
5. BDA writes results to S3 output
6. Result processor Lambda extracts fields, updates DynamoDB
7. Metrics emitted to SQS → metrics processor → S3 (Parquet) → Glue

### Auth Model

- **API clients**: API-Key header (DynamoDB-backed, tenant-scoped)
- **Admin users**: Cognito JWT (super-admin or tenant-admin roles)
- **Dual auth**: Endpoints support both via [`get_user_context_with_fallback`](documentai-api/src/documentai_api/utils/auth.py)
